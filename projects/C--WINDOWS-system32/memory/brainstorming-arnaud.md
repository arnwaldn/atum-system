# Brainstorming Arnaud — Preferences de travail (2026-03-07)

## Profil
- **Niveau**: Debutant en developpement, en apprentissage
- **Role ATUM**: Cofondateur, touche a tout (pas de role fixe)
- **Objectif**: Construire des produits concrets pour ATUM
- **Langue**: Explications en francais, code en anglais

## Comment travailler avec Arnaud

### Rythme et autonomie
- Valider chaque etape importante AVANT de continuer
- Ne JAMAIS faire des choses sans demander — il veut rester aux commandes
- Sessions longues et focalisees sur un seul projet

### Communication
- **Detaillee**: presenter les options, les trade-offs, pour qu'il comprenne et decide
- **Franche**: dire les problemes directement, sans enrober
- **Pedagogique**: expliquer les concepts techniques avec des analogies du quotidien
- Expliquer les termes techniques la premiere fois qu'ils apparaissent

### Apprentissage
- Apprend le mieux par des analogies avec le quotidien
- L'apprentissage vient en construisant, pas en theorie

### Resultats attendus
- Fonctionnel ET beau — les deux comptent autant
- Design: parfois il a une idee precise, parfois il fait confiance
- Quand il a une idee → lui demander exemples/screenshots
- Quand il n'en a pas → proposer quelque chose de moderne

### Qualite et tests
- Toujours tester — la fiabilite est prioritaire
- Montrer les resultats des tests concretement

### Quand ca bloque
- Continuer a chercher jusqu'a trouver, meme si c'est long
- Mais etre honnete si on tourne en rond

### Environnement
- Mix terminal (Claude Code) + navigateur pour voir le resultat
- Donner les URLs ou instructions pour verifier visuellement

## Mode de collaboration (2026-03-08)

### Avant de coder
- **Expliquer AVANT de coder** : presenter ce qu'on va faire et pourquoi, PUIS coder
- **Confirmer le scope** : s'assurer qu'on est d'accord sur ce qui va changer avant de toucher au code
- Ne pas se lancer directement dans le code sans contexte

### Autonomie adaptative
- **Simple** (typo, petit fix, config) → faire directement, sans demander
- **Important** (nouvelle feature, refactoring, architecture) → expliquer, valider, puis executer
- Le niveau d'autonomie depend de la complexite, pas du type de tache

### Irritants a eviter
- Les erreurs en boucle (si ca echoue 2 fois → changer d'approche)
- Le bruit dans le terminal (resolu via NTFS lock sur shell-snapshots)

## Resume en une phrase
Arnaud est un debutant qui veut construire des vrais produits, avec franchise, pedagogie, validation a chaque etape, et un resultat qui soit a la fois solide et beau.
